> Companion to `V4_0_2_BUG_TRIAGE_2026-05-28.lmm.md`. That journal
> identified the four bugs the user surfaced in screenshots of the
> live v4.0.2 dev build. This one documents the actual fixes shipped

## Phase 1 — RAW

After v4.0.2 shipped, the user launched the dev build and surfaced
four bugs via screenshots. All four are now fixed in v4.0.3.

### Bug 3 (highest priority) — PTY resize crash modal dialog

**Symptom**: JavaScript error popup at the user with stack
`WindowsPtyAgent.resize → WindowsTerminal.resize → PtyManager.resize
→ PtyRegistry.resize → IpcMainImpl`.  Triggered by renderer firing
resize events to a PTY whose underlying process had already exited.

**Root cause**: `PtyManager.spawnWithPty` registers `onExit` which
sets `this._pid = 0` but did NOT clear `this.ptyProcess`. The
`resize()` method's guard is `if (this.ptyProcess)`, which stayed
truthy after exit — so the call went through to node-pty and threw.

**Fix**: `src/main/pty-manager.ts`
- `onExit` handlers in both `spawnWithPty` and `spawnWithChildProcess`
  now null out the handle.
- `resize()` has a defensive `try/catch` that nulls the handle on
  any throw (covers a race where the conpty handle dies between
  the truthy-check and the call).
- `src/main/pty-registry.ts` `resize()` also wrapped in `try/catch`
  as defense-in-depth.

### Bug 1 — Claude (Chat) yellow diagnostic invisible

**Symptom**: Claude (Chat) tab spawns, prints `prompt argument when
using --print`, then `[process exited with code 1]`. The yellow
hint introduced in v4.0.2 (`EmbeddedTerminal` fast-exit detector
that suggests `claude --version` and the npm upgrade) was NOT shown.

**Root cause**: The diagnostic at `EmbeddedTerminal.tsx:136` is gated
on `profile === 'api.anthropic.claude-chat'`. But `ModelsPanel.tsx`
line 831 mounted `<EmbeddedTerminal>` without passing `profile` at
all, so it was always `undefined` in the Models-panel embed (the
surface used for the in-panel Running view).

**Fix**: `src/renderer/components/models/ModelsPanel.tsx`
- Pass `profile={running.find(r => r.paneId === selectedRunningPaneId)?.modelId}`
  so the diagnostic gates correctly on Claude (Chat) and the generic
  fast-exit fallback (line 145-149) fires for other model profiles.

### Bug 4 — Commands panel empty-state had no actionable affordance

**Symptom**: When the active tab is Claude (Chat), the Commands
panel shows only the empty-state text "Stream-JSON mode — type
your message in the composer; slash commands are not processed in
this mode." with no way to act on it.

**Fix**: `src/renderer/components/commands/CommandsPanel.tsx`
- Added a CTA banner at the top of the panel rendered when
  `family === 'claude-chat' && onSpawnPlainClaudeTab` is provided:
  "+ Switch to a plain Claude tab".
- Wired through `RightPanel` in `App.tsx` to the existing
  `handleNewClaudeTab` callback so the CTA spawns a Claude tab
  with the same semantics as the `+` button in TerminalTabs.
- Used the longhand `borderWidth`/`borderStyle`/`borderColor`
  triplet on the button's `style` prop to avoid React's
  shorthand-mixed-with-longhand reconciler warning (which the
  initial draft tripped via `border: '...'` + mouseEnter mutating
  `borderColor`).

### Bug 2 — Curated research "✓ Launched" downstream

**Symptom**: User reported that the bartowski Llama-3.2-3B
abliterated Q4_K_M Import card showed "✓ Launched" but the
spawned tab presumably exited fast (Ollama not running, or pull
failed, or the model PTY rejected something).

**Fix**: The same `profile`-prop fix in Bug 1 means the generic
fast-exit diagnostic at `EmbeddedTerminal.tsx:145-149` now fires
for any non-claude profile that exits within 3s with non-zero
code — covering the bartowski case. The user will now see the
yellow hint suggesting they try the same command in a regular
terminal to see Ollama's actual error.

## Phase 2 — NODES

- **PTY lifecycle**: the most subtle of the four bugs because
  it required understanding that `ptyProcess` is a held handle
  that doesn't auto-clear on exit. The fix is minimal but the
  diagnosis took finding the gap between `onExit` setting `_pid=0`
  and `resize`'s `if (this.ptyProcess)` guard.
- **Diagnostic visibility**: was a missed prop. v4.0.2 added the
  detector but didn't audit every call site. The CDP audits don't
  exercise model spawn → exit lifecycles, so they couldn't catch
  this either. (Punt for v4.1: add a PTY lifecycle smoke harness.)
- **Commands panel polish**: scope creep risk was real (could be
  argued the whole Claude (Chat) UX needs rethinking), but a
  single CTA button is the minimum-viable fix and matches the
  bug-fix-only scope of v4.0.3.

## Phase 3 — REFLECT

- The PTY crash class is exactly the kind of bug the user sees
  *during real use* that scripted audits miss. The 132/132 number
  is honest for the surface it covers; a long-running terminal +
  PTY lifecycle audit is the right addition for v4.1.
- Bug 1's root cause (missing prop) is a one-line fix that v4.0.2's
  scope SHOULD have caught with a `EmbeddedTerminal usages` audit.
  Adding a lint rule or audit check that EmbeddedTerminal always
  gets `profile` when mounted under Models or Tabs would prevent
  this regression.
- The user's pattern has been "no restrictions on uncensored models",
  but the Claude (Chat) profile is a separate concern — it's a
  CLI-feature-gating issue, not a content one. Keeping the diagnostic
  rather than blocking the entry is the right balance.
